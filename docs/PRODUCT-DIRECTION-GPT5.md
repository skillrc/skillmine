# OpenCode Local Asset Manager — 产品方向文档（GPT-5 草案）

## 1. 文档目的

本文档用于收敛当前项目的产品方向，不讨论具体实现细节，不进入代码设计，不展开兼容层细节。

目标只有一个：明确这个项目接下来到底要成为什么，以及什么暂时不做。

---

## 2. 核心结论

这个项目不应继续被定义为“一个 create-skill 工具”或者“一个从远程仓库安装 skill 的工具”。

更准确的方向应该是：

> **一个以 OpenCode 为核心、本地优先、私有优先的 AI 资产管理器。**

它服务的对象不是大众用户，也不是技能市场消费者，而是：

- 重度使用 OpenCode 的个人用户
- 希望沉淀内部 AI 工作流的团队
- 会持续创建、修改、重组自己资产的人

它的价值不在于“拿别人的东西一键安装”，而在于：

- 持续创建自己的资产
- 对已有资产进行深度改造
- 用稳定、清晰、可配置的方式管理本地 AI 能力体系
- 将这些本地资产投送到 OpenCode 的运行环境中

---

## 3. 问题重述

当前方向存在几个根本问题：

### 3.1 方向过度偏向远程安装

如果项目把 GitHub、远程仓库、公开分发作为核心起点，那么它会默认假设用户主要在“消费别人写好的 skill”。

但当前真实使用场景并不是这样。

对于重度用户来说，更常见的行为是：

1. 自己创建 skill
2. 长期维护自己的 skill
3. 拿到外部 skill 后进行改造
4. 最终形成私有资产体系

所以“远程安装优先”会把产品重心拉偏。

### 3.2 创建路径与运行路径没有清晰分层

如果一个工具把“项目源码目录”和“运行时数据目录”混在一起，或者默认把创建出来的 skill 放到不直观的位置，用户就会很难长期维护。

对于作者型用户，源码目录必须：

- 可见
- 明确
- 适合版本管理
- 适合人工阅读和修改

这也是为什么 `~/Project/Skills` 这样的目录更符合真实使用习惯，而 `~/.local/share/...` 更适合作为运行时数据目录。

### 3.3 概念仍然停留在“skill 工具”层面

当前项目如果只围绕 skill 来建模，会越来越难扩展。

因为真正需要管理的不只是 skill，还包括：

- command
- agent
- model profile

这些本质上都属于 OpenCode 生态中的“可配置本地资产”。

## 4. 新的产品定义

### 4.1 产品名称层面的理解

这个项目更像一个：

- **OpenCode Asset Manager**
- 或者 **OpenCode Workspace Manager**
- 或者 **OpenCode Personal/Team AI OS 的配置层**

无论最终命名如何，产品定义应该统一为：

> 管理 OpenCode 本地资产的创建、组织、安装、启用、同步与审计。

### 4.2 管理对象

v1 及后续都应围绕以下资产类型建模：

1. **Skill**：知识与流程资产
2. **Command**：动作入口资产
3. **Agent**：角色与执行单元资产
4. **Model Profile**：模型配置与模型组合资产

这些对象不应分别作为彼此无关的小工具，而应被纳入同一个统一系统。

---

## 5. 产品原则

以下原则建议作为项目的长期约束。

### 5.1 OpenCode-first

当前阶段只以 OpenCode 为唯一支点。

不提前为 Claude 或其他命令行 AI 工具设计适配层，不在一开始就追求跨平台抽象。

### 5.2 Local-first

优先支持本地创建、本地安装、本地管理、本地修改。

远程来源、公开仓库、GitHub 安装都不是当前阶段的核心能力。

### 5.3 Private-first

默认假设用户的资产是私有知识与私有工作流。

系统应该首先服务“个人长期积累”和“团队内部沉淀”，而不是公共发布。

### 5.4 Source / Install / Data 分离

必须严格区分以下三类路径：

- **Source**：源码项目目录
- **Install**：运行时安装目录
- **Data**：运行时可变数据目录

这是整个系统是否“顺手”的核心。

### 5.5 Configurable by default

必须有默认路径，但所有核心路径都应可配置。

“可配置”不是锦上添花，而是这个产品存在的理由之一。

### 5.6 Modification-friendly

产品不应该假设“安装后尽量不要动”。

相反，它应该默认支持：

- 安装后继续修改源码
- 重新安装
- 对外部资产进行本地改造
- 维护多个版本或多个变体

---

## 6. 三层路径模型

这是本项目最重要的基础模型。

### 6.1 Source Path

用于存放真正由用户维护的项目源码。

它应该是用户最常接触的目录，适合：

- 手工编辑
- Git 管理
- 长期维护
- 项目化组织

建议方向：

```text
~/Project/OpenCode/skills
~/Project/OpenCode/commands
~/Project/OpenCode/agents
~/Project/OpenCode/models
```

如果继续沿用现有习惯，也可以先从：

```text
~/Project/Skills
```

开始。

### 6.2 Install Path

用于存放 OpenCode 运行时真正读取的内容。

例如：

```text
~/.config/opencode/skills
```

未来还应有与 command、agent、model profile 对应的运行路径。

### 6.3 Data Path

用于存放运行中的可变数据、缓存、索引、知识状态等。

例如：

```text
${XDG_DATA_HOME:-~/.local/share}/opencode/...
```

`~/.local/share/...` 应该优先属于这个层，而不应成为主要源码目录。

## 7. 项目当前最合适的产品边界

### 7.1 v1 应该做什么

当前最适合的范围是：

#### Skill 全流程管理

- 创建
- 本地项目组织
- 安装到 OpenCode
- 列表与状态查看
- 重新安装
- 启用/停用
- 元数据管理

#### Command 全流程管理

- 创建
- 本地组织
- 安装
- 管理命名与分类
- 查看和启停

#### Agent 全流程管理

- 定义 agent 项目
- 管理 prompt、角色、边界、权限相关配置
- 安装与同步到 OpenCode
- 列出与审查 agent 配置

#### Model Profile 全流程管理

- 定义一组模型配置
- 支持多套模型方案
- 与 OpenCode 配置同步
- 允许不同场景切换不同 profile

### 7.2 v1 不应该做什么

以下内容建议明确排除：

- GitHub 远程安装
- 技能市场
- 公共分享平台
- 一键消费外部 skill 的流程优化
- Claude 兼容层
- 其他 CLI 工具适配层
- 复杂的跨生态抽象

这些不是永远不做，而是当前阶段不做。

## 8. 建议的产品主线

当前项目的主线不应再是：

> “如何安装一个 skill”

而应改成：

> **如何持续创建、沉淀、改造、组织并部署自己的 OpenCode 本地资产体系。**

这条主线的好处是：

- 能解释为什么需要创建器
- 能解释为什么需要安装器
- 能解释为什么需要配置系统
- 能解释为什么需要模型管理
- 能解释为什么需要 agent 管理

它能把项目从“工具集合”变成“一个完整系统”。

---

## 9. 远程安装为何应暂时撤下

基于当前目标，远程安装不是增强项，反而是干扰项。

### 9.1 它会把注意力从“创作与管理”拉向“分发与消费”

但当前真实用户并不主要消费公共资产，而是创作和改造私有资产。

### 9.2 它会提前引入一整套不必要复杂度

例如：

- 仓库来源可信度
- 版本兼容
- 元数据兼容
- 下载与更新策略
- 安装后改造与回写问题

这些问题会显著稀释当前阶段最需要解决的核心体验。

### 9.3 它会削弱产品的身份清晰度

当前最清晰的身份是：

> 面向个人与团队私有化 AI 工作流沉淀的本地管理器。

而不是：

> 一个公共插件安装器。

---

## 10. 对 agent 的理解要求

用户目前对 agent 的配置方式还不够清楚，因此 agent 应被视为一个必须先调研清楚的对象，而不是立即平铺到功能设计里。

当前可以先把 agent 看作：

- 一个角色单元
- 一个 prompt 载体
- 一个模型绑定点
- 一个工具权限边界
- 一个可被系统调度的执行对象

这意味着 agent 的管理不会等同于 skill 的管理。

它未来至少会涉及：

- agent 元数据
- prompt 文件布局
- model 绑定关系
- 工具权限定义
- 安装位置
- 运行时加载方式

因此，agent 在战略上要纳入产品范围，但在设计上需要单独做一轮事实调研。

---

## 11. 对 model profile 的理解

模型配置不应被看作零散设置，而应被看作“可切换的能力档案”。

也就是说，模型配置应该支持：

- 多套方案并存
- 针对不同任务/阶段的切换
- 与 OpenCode 配置同步
- 与 skill / command / agent 的组合关系

从这个角度看，模型不是附属功能，而是本地 AI 工作台的一部分。

---

## 12. 建议的系统视角

这个项目最终不只是“创建 skill 的工具”。

它更像一个：

### 12.1 作者工作台

让用户能够持续创作和修改自己的资产。

### 12.2 资产注册表

知道自己到底有哪些 skill / command / agent / model profile。

### 12.3 安装与同步系统

把本地资产稳定投送到 OpenCode 的运行环境。

### 12.4 本地 AI 组织层

帮助用户把原本分散的 AI 配置、指令、角色、模型整合成一个长期可维护的体系。

---

## 13. 最终建议

当前最合理的方向收敛如下：

### 13.1 产品定义

把项目定义为：

> **OpenCode-first、本地优先、私有优先的 AI 资产管理器。**

### 13.2 当前范围

只考虑 OpenCode，不考虑 Claude，不考虑其他 CLI。

### 13.3 当前重点

围绕 4 类资产建立统一心智模型：

- skill
- command
- agent
- model profile

### 13.4 当前裁剪

暂时撤下远程安装和 GitHub 仓库导向功能，把所有注意力放在本地工作流上。

### 13.5 成功标准

这个项目第一阶段是否成功，不看“能不能装别人写的 skill”，而看：

- 用户是否能在自己喜欢的项目目录下创建资产
- 用户是否能稳定安装到 OpenCode
- 用户是否能持续改造和维护这些资产
- 用户是否能管理多套模型与配置
- 用户是否感觉这个系统真正顺手

如果这几点成立，这个项目才算真正找到方向。
